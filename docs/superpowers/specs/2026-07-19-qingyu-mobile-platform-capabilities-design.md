# QingYu 移动平台能力设计

## 状态

本设计已于 2026-07-19 通过交互确认。它描述如何让现有 Tauri/React 应用真正运行于 Android 与 iOS，同时保持已经批准的 Compact 移动端产品模型。

本设计是以下规范的后续实现约束，而不是新的产品分支：

- [QingYu Compact 移动端设计](./2026-07-17-qingyu-compact-mobile-design.md) 定义移动端的信息架构、固定工作区、自动保存、同步与设置范围；
- [QingYu Compact Runtime Fixes Design](./2026-07-18-qingyu-compact-runtime-fixes-design.md) 定义无工作区状态与全屏页面隔离；
- [Complete Application Proxy Removal Design](./2026-07-19-complete-network-proxy-removal-design.md) 定义应用级网络代理的完整移除，移动端不得重新引入相关界面或请求字段。

若本文没有明确修改某项 Compact 行为，以上既有规范继续生效。

## 背景与证据

当前 Compact 界面和受管工作区逻辑已有自动化覆盖，但前端始终装配完整 `desktopRuntime`，Rust 入口也始终注册桌面菜单、多窗口、窗口状态、更新器、进程与桌面文件打开能力。浏览器缩窗能够验证界面，不能证明 Android 或 iOS 原生运行时可用。

真实移动编译已经暴露两类阻塞：

1. `arboard` 没有当前目标所需的 Android 后端；
2. 临时排除 `arboard` 后，Android 与 iOS 都继续在桌面菜单、窗口状态和窗口装饰 API 上产生编译错误。

这说明移动端不能通过给桌面入口增加少量条件判断完成。应用需要共享产品核心，同时在 TypeScript 运行时、Rust 命令注册、插件依赖和 Tauri capability 四个边界上明确拆分桌面与移动能力。

## 目标

- 同一套 React、编辑器、文件模型、项目配置和同步核心同时服务桌面与移动端；
- Android 与 iOS 只编译、注册并授权移动端实际使用的能力；
- 首版移动端完成编辑、文件管理、历史恢复、图片导入、本地模式与现有远程同步闭环；
- 不可用的桌面功能不以失效按钮或运行时崩溃的形式暴露；
- Android 和 iOS 的原生工程能够重复生成、编译、安装和执行核心验收流程；
- 桌面宽屏行为与现有同步、冲突、删除传播和 checkpoint 语义不变。

## 非目标

首版不包含：

- 桌面全部功能的一比一移植；
- 多工作区、目录切换、App 内重置或清空数据；
- 移动端独立的同步协议、后台常驻同步服务或新的冲突解决器；
- 任意附件导入、Markdown 文件导入、系统文件关联、分享目标或“用 QingYu 打开”；
- Pandoc、HTML/PDF 导出、模板、更新器、原生日志目录、终端命令或桌面窗口管理；
- 系统字体枚举、自定义字体文件选择或移动端专属字体管理；
- 应用级网络代理或兼容字段；
- 为移动端重写编辑器、同步引擎或文件树业务逻辑。

## 总体架构

### 一个应用，两套平台适配器

共享应用层继续持有文档、编辑器、文件树、受管工作区、项目配置、同步协调器和设置状态。平台差异只通过运行时接口与原生适配器进入：

```text
共享 React 产品层
文档 / Milkdown / 文件树 / 设置 / 项目配置 / 同步
                       │
              AppRuntime 能力边界
                 ┌─────┴─────┐
                 │           │
        Desktop Runtime   Mobile Runtime
        桌面窗口与菜单      移动文件、图库与返回键
                 │           │
        Desktop Rust      Mobile Rust
        命令与插件集合      命令与插件集合
```

不得在共享组件中散布 Android、iOS、macOS、Windows 或 Linux 判断。平台选择在启动阶段完成一次，组件只读取明确的能力或调用已注入的方法。

### 前端运行时选择

应用启动时通过 Tauri OS 插件识别 `android` 或 `ios`，然后装配 `mobileRuntime`；其他原生平台装配 `desktopRuntime`。浏览器运行时继续使用 Web 适配器，桌面浏览器缩窗只承担 Compact UI 验证，不冒充移动原生能力。

`AppRuntime` 可以继续保持稳定的共享形状，但必须满足：

- 移动端使用移动实现或显式的不支持实现，不能导入会调用桌面命令的模块；
- 所有用户可见入口由能力控制，不允许仅在点击后才发现底层命令不存在；
- 能力命名表达产品动作，例如导出、更新、多窗口、图片导入，而不是让组件推断某个平台应支持什么；
- 已批准移除的网络代理能力从共享契约中删除，不作为移动端 false 标志保留。

### Rust 编译与注册边界

Rust 入口按目标平台拆分模块、依赖、插件与 `invoke_handler`：

- 共享核心保留 Markdown 读写、受管工作区、项目配置、文件历史、监听、WebDAV/S3 同步、冲突处理和 Web 图片下载；
- 桌面目标保留原生菜单、上下文菜单、多窗口、窗口状态恢复、窗口装饰、外部文件打开、Pandoc、桌面导出、更新器和进程重启；
- 移动目标只注册共享核心以及移动端图片导入、外部链接、生命周期和返回键所需的适配；
- `arboard`、窗口状态、单实例、更新器、`macos-private-api` 以及只被桌面命令使用的依赖或 feature 采用桌面 target 配置或桌面 `cfg`；
- 仅给函数体加条件分支不够：移动编译单元不得解析桌面专用类型、trait 或 builder 调用。

共享命令的请求和返回结构保持跨平台一致；桌面专用命令不进入移动端 handler。前端移动适配器也不得 invoke 这些未注册命令。

当前文件监听继续使用共享 `notify` 实现，不因目标是移动端就预先更换库。Android/iOS 编译与设备测试必须证明它能观察同步和应用内部写入；如果设备事件不可靠，移动适配器应在已知的文件操作与同步完成后主动使文件树失效并重新加载，而不是引入常驻轮询或另一套同步状态。

### Capability 与最小权限

桌面继续使用现有 capability。移动端建立独立 capability，只授权主移动窗口和首版功能所需权限：

- App 路径、事件、受管工作区文件读写；
- `store`、`dialog`、`fs`、`os`、`log` 与 `opener` 的必要子权限；
- 图片选择和读取系统返回 URI 所需的移动权限；
- 同步所需网络访问。

移动 capability 不包含菜单、窗口最小化/最大化、创建额外窗口、进程重启、更新器或任意 Shell 权限。权限范围应尽量限定于应用内部工作区及由系统选择器授予的 URI，不向整个设备存储申请宽泛访问。

Android 和 iOS 生成工程属于可重复移动构建的一部分。首次 `tauri android init`/`tauri ios init` 产生的项目骨架，以及项目要求的 manifest、entitlement、Info.plist 与构建配置变更，应纳入版本控制；各生成工程自身 `.gitignore` 排除的构建产物、缓存和本机配置继续忽略。不得把签名私钥、开发者账户、真实同步凭据、测试端点或本机绝对路径提交到仓库。

## 功能取舍

### 保留并共享

首版移动端必须保留：

- WYSIWYG Markdown 编辑、撤销重做、选择与基础格式；
- App 私有持久目录中的文件/文件夹创建、重命名、移动、删除、搜索和历史恢复；
- 全屏文件目录、全屏设置页及编辑器状态保留；
- 打开上次文档；没有可恢复文档时显示编辑器欢迎空状态；
- 现有 WebDAV/S3 配置、连接测试、手动同步、前台同步触发、删除传播、checkpoint 与冲突副本；
- 本地模式，不要求用户必须配置同步；
- 主题、语言以及适用于移动编辑器的设置；
- Markdown 中的本地图片与远程图片显示。

固定工作区位于操作系统提供的 App 私有持久数据目录，不能使用可能被系统清理的临时缓存目录。应用同时只管理这个目录，不提供选择、切换、重置或删除云端内容的入口。

### 用移动实现替换

以下能力保留产品目的，但替换桌面实现：

| 产品动作 | 移动实现 |
| --- | --- |
| 打开外部网页 | 官方 `tauri-plugin-opener`，不执行 `xdg-open`、`open` 或 `start` 子进程 |
| 从设备添加图片 | 官方 `tauri-plugin-dialog` 选择图片，使用 `tauri-plugin-fs` 读取系统返回的 URI，再复制进工作区 |
| 新建/重命名输入 | App 内模态输入组件，保持当前“先输入名称再创建”的语义，不使用 `window.prompt` |
| 删除/放弃更改确认 | App 内确认或移动端兼容的 dialog 接口，按钮文案与危险操作层级明确 |
| 剪贴板文本 | 优先依赖 WebView 的系统选择菜单与 Web Clipboard；只有真机验证证明不足时才引入官方剪贴板插件 |
| Android 系统返回键 | Compact 导航栈非根页面先返回上一层；编辑器根页面交还系统处理 |
| 生命周期保存 | 保留 1.5 秒自动保存，并在 `visibilitychange`/`pagehide` 前台退场时提交待保存内容 |

### 移动端隐藏并编译排除

移动端不显示也不注册：

- 应用菜单、桌面上下文菜单和菜单快捷键；
- 新窗口打开、设置独立窗口、多窗口恢复、窗口状态、最小化、最大化、拖动标题栏和桌面装饰；
- 打开所在文件夹、用系统默认应用打开任意附件或打开日志目录；
- Markdown/文件夹系统选择、最近目录和外部工作区切换；
- Pandoc 检测与安装、HTML/PDF 等桌面导出、Shell 命令；
- 桌面应用更新检查、下载、安装与进程重启；
- 系统字体枚举与字体文件选择；
- 移动端不适用的桌面诊断项和设置分类。

设置页只渲染当前运行时支持的分类和字段。被排除功能不能保留灰色占位项，也不能在设置导入后重新出现。

## 图片导入设计

首版移动导入只接受系统图库或文件选择器返回的图片，不接受 Markdown、PDF、音视频或一般附件。

数据流如下：

1. 用户在编辑器触发添加图片；
2. 系统选择器按现有编辑器支持的图片类型进行过滤；
3. 用户取消时不显示错误，也不修改文档；
4. Android 返回的 `content://`、iOS 返回的文件 URI 或沙箱副本由 `plugin-fs`/移动适配器读取，不能先转换为 Rust `PathBuf` 再沿用桌面命令；
5. 适配器验证返回内容确实是受支持图片，并取得可用扩展名或 MIME 信息；
6. 使用现有安全命名规则将图片原子写入固定工作区的资源目录；重名时生成不覆盖已有文件的名称；
7. 只有写入成功后才向编辑器插入工作区相对 Markdown 引用；
8. 写入失败时不留下半文件、不插入失效引用，并显示可理解的原因。

如果系统选择器提供临时副本，完成工作区复制后按插件约定释放或清理临时资源。同步仍把已复制进工作区的图片当作普通项目文件处理，无需为移动图片增加同步协议。

通过同步进入工作区的非图片附件仍可出现在文件树中，以免误导用户认为文件丢失；点击时显示“此移动版本不支持打开该附件”的明确提示，不尝试调用系统 opener。

## 启动、生命周期与同步

### 启动

1. 初始化最小移动插件集合和持久 Store；
2. 解析或创建固定 App 私有工作区；
3. 加载项目配置与同步状态；
4. 恢复上次打开且仍存在的文档；
5. 无可恢复文档时进入编辑器欢迎空状态；
6. 受管工作区尚未就绪时保持既有 gate，不创建第二个工作区。

应用升级不得移动或清空工作区。系统数据被用户从系统设置中清除后的首次启动按全新安装处理，这不等于 App 提供重置功能，也不会主动修改云端。

### 前后台

移动端不建立常驻后台同步服务。编辑、保存和同步都沿用前台模型：

- 编辑过程中继续使用已批准的 1.5 秒自动保存；
- 显式保存与页面隐藏会刷新待保存内容；
- 1.5 秒快速保存和页面隐藏补存只写本地，不调用 `notifyDocumentSaved`，因此不产生项目同步的 `save` 触发；
- 用户显式保存继续走现有保存包装器，并只在现有 `autoSyncOnSave` 配置启用时产生 `save` 触发；
- 回到前台时按现有启动/恢复触发检查同步状态；
- 操作系统强制终止前无法保证异步任务完成，因此不能把唯一保存机会推迟到退出事件。

### 同步

WebDAV/S3 引擎、manifest、checkpoint、删除传播和冲突副本算法不产生移动分叉。移动端只改变入口布局与平台传输条件。

配置不受支持或连接失败时显示明确错误和原因，最多提供一个“前往同步设置”按钮。用户可直接修改配置，不要求重置 App。凭据、密钥和授权头不得进入 UI 错误、日志或诊断报告。

本地 MinIO 测试通过环境变量或本机私密配置注入 endpoint、access key、secret key 与 bucket，仓库中的测试、fixture、截图和文档不得记录这些值。

## 网络与系统配置

移动端需要普通互联网权限以访问 WebDAV、S3、远程图片和外部网页。它不恢复应用级代理设置。

局域网 HTTP 服务必须分别验证：

- Android debug 构建具备网络权限以及测试所需的 cleartext 策略；release 默认不应因测试方便而全局放宽明文流量；
- iOS 为访问局域网服务提供清晰的本地网络用途说明，并按最小范围配置 ATS/local-networking；
- 权限拒绝、系统弹窗拒绝、DNS/超时、TLS、认证和 endpoint 格式错误都映射为用户可理解的同步错误。

正式产品配置仍应优先使用 HTTPS。局域网明文放行是开发和指定部署需求，不是默认安全策略。

## 错误处理

错误展示遵循“说明发生了什么、给出原因、最多一个下一步动作”：

- 工作区创建或访问失败：阻止进入编辑器，说明系统存储错误并允许重试；
- 图片权限拒绝：说明需要照片/文件访问，允许再次触发系统选择；
- 图片格式不支持：说明仅支持图片，不创建文件；
- 图片复制失败或空间不足：保持文档不变并允许重试；
- 同步配置或连接错误：显示现有 provider 错误，唯一快捷操作是进入同步设置；
- 非图片附件点击：显示不支持提示，不调用原生打开命令；
- 外部链接无法打开：保留当前页面，显示 opener 失败原因；
- 后端未注册命令属于构建缺陷，不能作为可恢复的用户错误交付。

所有错误都应通过共享错误边界或对应全屏页面显示，避免原生崩溃、无限加载和只写日志不反馈。

## 测试设计

### 自动化测试

前端测试必须覆盖：

- Android/iOS 选择 `mobileRuntime`，桌面选择 `desktopRuntime`，浏览器选择 Web Runtime；
- 移动能力矩阵不会渲染桌面菜单、多窗口、导出、更新器、Pandoc、字体与系统目录入口；
- App 内命名输入的创建、取消、空值、重名与错误路径保持现有语义；
- 图片选择取消、URI 读取、格式拒绝、复制失败、重名、成功插入和失败不插入；
- Android Back 在文件、设置、移动和同步页面逐层返回，在根编辑器不截获系统行为；
- 页面隐藏刷新保存，恢复时打开上次文档或欢迎空状态；
- 不支持附件提示与外部 URL opener 的成功/失败路径；
- 同步错误只提供一个进入同步设置动作，秘密信息不会泄漏。

Rust 测试必须覆盖：

- 移动与桌面 cfg 下注册的命令/插件集合符合各自白名单；
- 受管工作区位于持久 App 数据目录且路径逃逸被拒绝；
- 图片目标命名、原子复制、失败清理和相对路径生成；
- 共享同步测试在未改变语义的情况下继续通过；
- Android/iOS 编译不再解析 `arboard`、菜单、窗口状态或其他桌面专用代码。

### 构建门禁

每次移动能力变更至少运行：

- `cargo test --manifest-path apps/desktop/src-tauri/Cargo.toml`；
- `pnpm test`；
- `pnpm typecheck:test`；
- `pnpm build`；
- Android debug 原生构建；
- iOS Simulator debug 原生构建。

真实 MinIO 可用时再运行 `pnpm test:s3-sync:live`，凭据只通过本机环境提供。Android 与 iOS 构建成功的标准是产出可安装制品或可运行 Simulator App，而不是仅完成前端 Vite 构建。

### 设备验收矩阵

Android 模拟器、iOS Simulator 至少验证：

- 首次启动与固定工作区创建；
- 新建、输入、自动保存、重启恢复、重命名、移动、删除与历史；
- 文件目录和设置全屏覆盖、编辑器不丢状态；
- 本地模式完整使用；
- WebDAV/S3 正常同步与手动同步；
- 错误 endpoint、错误凭据、不可达服务、离线、超时与冲突副本；
- 图片选择、复制、显示、重启后显示与同步后显示；
- 非图片附件、不支持格式、权限拒绝与空间不足；
- 外部链接、系统返回键、前后台切换和强制关闭后的恢复。

真机补充验证模拟器不能可靠证明的行为：软键盘/输入法、中文输入、系统文本选择与剪贴板、图库权限、Android `content://`、iOS 沙箱 URI、本地网络权限、低内存恢复、物理返回键/手势及实际文件持久性。

桌面回归必须验证宽屏界面、桌面原生菜单、多窗口、导出、更新与现有项目目录行为没有因移动拆分而退化。浏览器缩至手机宽度继续作为快速 Compact UI 回归手段，但不能替代原生设备门禁。

## 实施顺序

1. 先建立前端 runtime 选择、产品能力矩阵和对应测试，不改共享业务语义；
2. 再拆分 Rust target dependencies、模块、插件、handler 与 capability，使 Android/iOS 空壳可以编译启动；
3. 接入受管工作区、App 内命名输入、外部链接、返回键和生命周期保存；
4. 接入 URI 驱动的图片导入并完成异常路径；
5. 运行仓库门禁、Android/iOS 编译、模拟器核心流程和真实 MinIO 测试；
6. 最后完成真机权限、键盘、图库、局域网与生命周期验收。

每一阶段保持桌面测试可运行。不得先复制同步引擎或编辑器形成移动分支，再尝试回收共享逻辑。

## 验收标准

设计完成实施后，应同时满足：

- Android 与 iOS 能从干净环境重复构建并启动；
- 移动端只展示被批准的首版功能，不存在点击后调用未注册桌面命令的入口；
- 用户无需配置同步即可完成本地编辑闭环；配置 WebDAV/S3 后沿用桌面同步和冲突行为；
- 图片能从图库/系统文件选择器复制进工作区并以相对引用插入，取消或失败不会污染文档；
- App 只使用一个持久内部工作区，不提供切换、重置或云端删除；
- 上次文档恢复、欢迎空状态、1.5 秒自动保存、全屏目录与设置、返回键和前后台保存符合既有 Compact 规范；
- 不支持的附件和同步配置给出简洁、明确、可行动的错误；
- 测试凭据、私有 endpoint 与签名材料不进入仓库；
- 仓库级测试、Rust 测试、类型检查、Web 构建、移动原生构建、模拟器核心流程与桌面回归全部通过。
